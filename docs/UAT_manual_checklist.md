## 2. 认证流程（Auth）

### 用例 2.1 用户注册
- **步骤**
  1. 点击 Register
  2. 输入用户名、邮箱、强密码并提交
- **预期**
  - 注册成功并进入已登录态（或提示成功后可登录）
  - Network 中 `POST /api/v1/auth/register` 返回 `201`

### 用例 2.2 用户登录
- **步骤**
  1. 点击 Login
  2. 使用刚注册账号登录
- **预期**
  - 登录成功，页面显示用户可用功能
  - Network 中 `POST /api/v1/auth/login` 返回 `200`
  - 后续请求带 `Authorization`（如实现为 token 持久化）

### 用例 2.3 当前用户信息
- **步骤**
  1. 登录后刷新页面
- **预期**
  - 登录态仍有效（或按产品设计要求重新登录）
  - `GET /api/v1/auth/me` 返回 `200`

## 8. 验收判定

- **通过标准**
  - 核心链路无阻断：注册/登录 → 角色 CRUD → 世界 → 模拟控制 → 关系查询
  - 关键接口返回码符合契约
  - 页面可用且无致命崩溃

- **失败标准**
  - 任一核心步骤出现 5xx 或页面不可恢复错误
  - 登录后核心功能无法继续（如无法创建角色/世界）

## 9. 常见故障与定位

### 9.1 注册/角色接口返回 500
- 常见原因：数据库未就绪（例如本地 PostgreSQL 未启动）
- 处理建议：
  - 启动 PostgreSQL，或
  - 用 SQLite 本地模式启动后端（开发态）

### 9.2 页面显示 Not authenticated
- 检查 `auth/login` 是否 `200`
- 检查 token 是否被前端持久化并随请求发送

### 9.3 WS 无实时更新
- 检查 `/api/v1/ws/{world_id}` 连接是否建立
- 检查是否已发送 `join_world`
- 检查模拟是否已启动
